# 사진 기반 자동 방문 감지 (Photo Visit Detection)

> 상태: **기획(Spec)** · 작성일 2026-07-16 · 대상 플랫폼: 네이티브 앱(iOS/Android, Capacitor)
> 선행 조건: 네이티브 앱 빌드 (웹/PWA로는 구현 불가 — 사유는 §9)

## 1. 개요

사용자는 **밥을 먹는 순간엔 앱을 켜지 않는다**. 대신 **음식 사진은 찍는다**. 사진에는 촬영 시각과 GPS가 남는다. 이 사진을 단서로, 앱이 **"어디서·언제 먹었는지"를 대신 기억**하고, 나중에 리뷰·식당 등록을 유도한다.

핵심 가치는 **미등록 식당의 발견**이다. 방에 이미 등록된 식당은 사진을 이미 올렸을 확률이 높다. 반면 **아직 기록하지 않은 새 식당의 사진**은 새로운 식당 등록과 리뷰를 낳고, 방 전원의 참여를 유도한다. 이것이 리텐션(재방문)과 콘텐츠 공급(공개 방·SEO)을 동시에 푸는 엔진이다.

## 2. 문제 정의 · 가설

- **문제**: 웹 방문 재방문율이 매우 낮음(30일 재방문 8명 수준). 원인은 "기록이 귀찮음" + "돌아올 트리거 없음".
- **가설**: 사용자가 찍은 음식 사진을 배치로 스캔해, 식당 좌표와 매칭되면 "다녀오셨죠? 기록할까요?" 푸시를 보내면 → (1) 기록 마찰이 사라지고 (2) 재참여 트리거가 생겨 리텐션이 오른다.
- **성공 지표**
  - 감지 후보 → 사용자 확정 비율(정밀도 프록시) ≥ 60%
  - 감지 경유 신규 식당 등록 수 / 전체 신규 식당 등록 수
  - 푸시 → 앱 오픈 → 리뷰 작성 전환율
  - 사진 스캔 활성 사용자군의 D7/D30 리텐션 vs 비활성군

## 3. 원칙 (설계 제약)

1. **GPS 없는 사진은 처리하지 않는다.** 위치 없는 추정은 오탐만 낳으므로 스킵. (사용자 확정 결정)
2. **음식 여부는 판별하지 않는다.** 대신 **"GPS가 식당 POI 위인가"로 대체**한다. 위치가 판별을 대신함. 한 장소에 사진이 여러 장 뭉쳐 있으면(체류) 신뢰도 가중.
3. **절대 자동 기록하지 않는다.** 감지는 후보(candidate)일 뿐, 반드시 사용자 **확인 후** 서버에 기록한다. 오탐으로 안 간 곳에 리뷰가 생기면 신뢰가 무너진다.
4. **프라이버시 우선.** 원본 사진·전체 위치 이력은 **기기를 떠나지 않는다.** 서버는 사용자가 확정한 결과만 받는다. 내 사진이 내 방문을 방 전원에 자동 폭로하지 않는다(§8).
5. **옵트인.** 사진첩 접근·스캔은 명시적 동의 후에만 동작한다.

## 4. 사용자 시나리오

### 4-1. 이미 방에 있는 식당 — 리뷰 유도
1. 내가 방의 식당 "○○파스타"에 다녀와 사진을 찍음.
2. 몇 시간 뒤 배치 스캔이 사진 GPS를 "○○파스타" 좌표와 매칭.
3. 그 방문 날짜에 **내 리뷰가 없으면** 로컬 푸시: "어제 ○○파스타 다녀오셨죠? 별점 30초면 끝".
4. 탭 → 사진·날짜·식당이 미리 채워진 리뷰 화면 → 저장.

### 4-2. 미등록 식당 — 발견 & 등록 (핵심 가치)
1. 방 어디에도 없는 새 식당에서 사진을 찍음.
2. 배치 스캔이 좌표를 **주변 음식점 POI**와 매칭 → "새 맛집 발견".
3. 로컬 푸시: "새로운 곳에서 찍으셨네요 — △△식당, 방에 추가할까요?"
4. 탭 → **어느 방에 추가할지 선택**(다중 방 소유 대응) → 식당 등록 + 리뷰.
5. 등록되는 순간 **기존 '식당 등록됨' 알림**이 방 멤버에게 발사(재사용) → 안 쓴 멤버 리뷰 유도.

### 4-3. 동행자 참여 루프
- 방문에 동행자를 태그(`RoomVisitParticipant`, 기존 모델)하면, **아직 리뷰 안 쓴 동행자**에게 "같이 간 ○○, 리뷰 남길래요?" 알림. 전원 참여 유도.

## 5. 시스템 아키텍처

```
[네이티브 앱 · 온디바이스]
  BGTaskScheduler(iOS) / WorkManager(Android)  — 몇 시간 주기
    1. 사진첩에서 lastScanCursor 이후 추가된 사진 조회 (GPS 있는 것만)
    2. (위치 ~50m) + (시간 ~2h) 클러스터링 → 방문 후보 생성
    3. 후보 좌표 매칭:
        a) 오프라인 1차: 동기화된 '내 방 식당 좌표'와 대조   ← API 0원
        b) 미매칭 시: OS 지오코더(무료) / 백엔드 POI 프록시로 주변 음식점 조회
    4. 매칭 결과별 로컬 알림 생성 (자동 기록 없음)
         - 방 식당 + 내 리뷰 없음 → 리뷰 유도
         - 신규 음식점 POI       → 발견/등록 유도
         - POI 아님 / 식당 아님   → 스킵
  ↓ 사용자 확정 시에만
[백엔드]  기존 등록/리뷰/알림 API 재사용 (§7)
```

- **감지·매칭은 전부 온디바이스.** 사진과 위치 이력은 서버로 보내지 않는다.
- **네트워크는 (a) 내 방 식당 좌표 동기화, (b) 신규 POI 조회 프록시, (c) 사용자 확정 시 쓰기** 3곳뿐.

## 6. 데이터 모델 (Prisma)

기존 재사용: `RoomRestaurant(latitude,longitude)`, `RoomVisit`, `RoomReview`, `RoomVisitParticipant`, `RoomNotification`.

신규/변경:

```prisma
model User {
  // ...
  photoScanEnabled   Boolean   @default(false)  // 옵트인
  photoScanConsentAt  DateTime?
}

// 사용자가 확정한 감지 방문 (분석·중복방지용, 온디바이스 후보는 저장 안 함)
model DetectedVisit {
  id            String   @id @default(cuid())
  userId        String
  restaurantId  String?  // 매칭/등록된 식당 (신규는 등록 후 채움)
  roomId        String?
  latitude      Float
  longitude     Float
  photoTakenAt  DateTime // 사진 촬영 시각(방문 추정)
  source        String   @default("photo") // photo | manual
  status        String   @default("confirmed") // confirmed | dismissed
  createdAt     DateTime @default(now())

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@index([userId, photoTakenAt])
}

// 좌표→POI 역지오코딩 캐시 (외부 API 호출 최소화)
model PoiCache {
  id         String   @id @default(cuid())
  geoHash    String   @unique // 반올림 좌표 키 (예: 소수 4자리)
  name       String
  category   String   // 카카오 카테고리 (FD6=음식점, CE7=카페 등)
  address    String
  latitude   Float
  longitude  Float
  fetchedAt  DateTime @default(now())
  @@index([geoHash])
}
```

- 온디바이스의 **후보(candidate)** 는 로컬 저장(SQLite/Preferences). 서버엔 확정분(`DetectedVisit`)만.
- `lastScanCursor`(마지막 스캔 지점)는 기기 로컬 저장 — 재스캔 방지.

## 7. API 설계

| Method | Path | 설명 |
|---|---|---|
| `PATCH` | `/users/me/photo-scan` | 사진 스캔 옵트인/아웃 (`photoScanEnabled`) |
| `GET` | `/users/me/room-restaurants/geo` | 내 방 식당 좌표 목록 `{roomId, restaurantId, name, lat, lng, myReviewDates[]}` — **오프라인 매칭용** |
| `GET` | `/rooms/poi/nearby?lat&lng&radius` | 주변 음식점 POI (카카오 프록시 + `PoiCache`) |
| `POST` | `/users/me/detected-visits` | 사용자 확정 감지 방문 기록 (분석·중복방지) |

- 식당 등록/방문/리뷰는 **기존 API 재사용** (`POST /rooms/:id/restaurants`, `.../visits`, `.../reviews`). 등록 시 기존 `createNotificationForRoom`이 멤버 알림 발사.
- 신규 엔드포인트는 얇게. 매칭 로직은 앱에.

## 8. 프라이버시 · 동의

- **옵트인 필수**: 최초 진입 시 "사진으로 방문을 자동 감지할까요?" 동의 화면. iOS는 **선택 사진 접근(제한 권한)** 지원 → "전체 허용" 강요 안 함.
- **원본 미전송**: 사진·위치 이력은 기기에만. 서버는 확정 결과만.
- **자동 폭로 금지**: 감지 즉시 방 전원에 방송하지 않는다. **감지자가 확인·등록한 뒤에야** 기존 등록 알림이 나간다. 즉 "누가 어디 갔다"를 본인 승인 없이 노출하지 않음.
- **동행자 알림**: 방문 참여자로 **명시 태그된** 사용자에게만. 임의 위치 상관관계로 타인을 엮지 않음.
- 개인정보처리방침에 사진첩/위치 이용 목적·범위·보관 명시.

## 9. 왜 웹 불가 (네이티브 필수)

| 필요 능력 | 웹/PWA |
|---|---|
| 사진첩 전체 열람·스캔 | ❌ API 없음 (유저가 고른 파일만) |
| 백그라운드 스캔 | ❌ 사진 접근 백그라운드 불가 (Periodic BG Sync도 사진 접근 X, iOS Safari 미지원) |
| 원본 EXIF GPS | ⚠️ iOS Safari 업로드 시 GPS 제거 |
| 웹 푸시 | ✅ 가능하나 트리거할 스캔이 없음 |

→ 사진첩 배치 스캔 + 원본 GPS + 백그라운드 = **네이티브 전용.** 이 기능이 곧 앱을 만드는 이유. (Capacitor 스캐폴드 기 존재)

## 10. POI 비용 최소화 전략

우리 볼륨(월 수십 방문)에선 외부 API 비용은 사실상 0이지만, 설계로 호출을 더 줄인다.

1. **오프라인 우선 매칭**: "내 방 식당" 케이스(재참여 핵심)는 동기화된 좌표와 온디바이스 대조 → **API 0**.
2. **OS 기본 지오코더**: iOS CLGeocoder / Android Geocoder는 무료·키 불필요. 주소·장소명 1차 확보.
3. **좌표 반올림 캐싱**(`PoiCache`, geoHash): 같은 자리 반복 방문은 캐시 히트 → 재호출 0.
4. **카카오/네이버 프록시는 "정확히 어느 식당" 필요할 때만** 보조 호출. (키는 백엔드 보관, 기존 프록시 패턴 재사용)

→ 실제 비용 부담은 API가 아니라 **네이티브 앱 개발·유지 공수**.

## 11. 기술 리스크 · 완화

| 리스크 | 완화 |
|---|---|
| 도심 밀집 오탐(한 건물 다수 식당, GPS ±30m) | 후보를 **여러 개 제시하고 사용자 선택**. 사진 클러스터·체류로 신뢰도 가중 |
| GPS 없는 사진 | **처리 안 함**(원칙 §3-1). 조용히 스킵 |
| 백그라운드 실행 타이밍 불확정(OS 재량) | "즉시" 대신 "당일/몇 시간 내" 기대치. 앱 포그라운드 진입 시에도 스캔 트리거 |
| 사진첩 권한 거부 | 옵트인·제한 권한 지원. 거부 시 기능 비활성(앱은 정상) |
| iOS 지오펜스/백그라운드 제약 | 지오펜스 대신 **사진 EXIF 배치** 방식이라 20개 제한 무관 |
| 프라이버시 우려 | 온디바이스 처리 + 확정 후 전송 + 명시 동의 |

## 12. 단계별 PoC / 로드맵

- **Phase 0 — 기획(현재)**: 본 문서.
- **Phase 1 — Android 포그라운드 PoC**: 앱 오픈 시 최근 사진 스캔 → 매칭 정확도 검증(백그라운드·푸시 없이). WorkManager·EXIF·GPS 유지가 안드에서 가장 수월·저렴.
- **Phase 2 — Android 백그라운드 + 로컬 푸시**: WorkManager 주기 스캔 → 로컬 알림.
- **Phase 3 — iOS**: BGTaskScheduler + 제한 사진 권한.
- **Phase 4 — 참여 루프**: 신규 식당 발견 → 방 등록 → 멤버 알림 → 동행자 리뷰 유도.

각 Phase 종료 시 §2 성공 지표로 진행 여부 판단(게이트).

## 13. 미해결 질문

- 신규 식당의 **방 자동 추천** 정확도(다중 방 소유 시) — Phase 4에서 사용자 최근 활동 방 기준 추천할지.
- 배치 주기 최적값(배터리 vs 신선도).
- POI 소스: 카카오 vs 네이버 vs OS 지오코더 — 한국 음식점 커버리지 비교 필요.
- 오탐 시 사용자 피드백을 매칭 신뢰도에 반영할지(학습 루프).
