# 코드 리뷰 가이드

PR 리뷰 시 아래 체크리스트를 기준으로 검토한다.

---

## P1: CRITICAL (즉시 수정 필요)

머지 차단. 반드시 수정 후 재리뷰.

| # | 항목 | 설명 |
|---|------|------|
| 1 | **null 참조 가능성** | DB 조회 결과 등 null 가능한 값을 체크 없이 사용 |
| 2 | **브레이킹 체인지** | Request/Response DTO 필드 제거 또는 타입 변경 |
| 3 | **N+1 쿼리** | 루프 내부 개별 DB 쿼리로 커넥션 풀 소진 위험 |
| 4 | **Race Condition** | 조회-업데이트 사이 트랜잭션 미사용 |
| 5 | **컴파일 에러** | duplicate identifier, 타입 에러 등 |
| 6 | **보안 취약점** | SQL Injection, XSS, 인증 우회 |
| 7 | **any/unknown 사용 금지** | `any`, `unknown` 타입 사용 절대 금지. 명확한 타입 정의 필수 |
| 8 | **NaN/nullable/falsy 검증** | NaN, nullable, undefined 참조 및 의도치 않은 falsy 값(`0`, `''`, `false`) 비교 검증 필수 |
| 9 | **DTO ↔ Controller ↔ Service 필드 동기화** | DTO에 필드 추가 시 Controller에서 Service로 전달하는 객체에도 반드시 포함. `whitelist: true`가 DTO에 없는 필드를 제거하므로 DTO-Controller-Service 3곳이 항상 일치해야 함 |
| 10 | **패턴 일괄 변경 시 전체 grep 검증** | 동일 패턴을 여러 곳 수정할 때(예: `img.url` → `toImageUrl(img.url)`) 반드시 grep으로 변경 전 패턴이 남아있는지 확인. 수정 후에도 누락 검증 필수 |
| 11 | **정수 검증 `@IsInt()` 강제** | 정수 필드에 `@IsNumber()` 사용 금지 — 소수점이 허용되어 DB 타입 오류 가능. 반드시 `@IsInt()` 사용 |

## 리뷰 프로세스

### 리뷰어 규칙
- PR은 최소 1명 이상의 Approve가 있어야 머지 가능
- P1 항목이 1개라도 있으면 Request Changes
- P2 항목만 있으면 코멘트 후 Approve 가능
- P3 항목은 Nit으로 남기고 Approve

### PR 작성자 규칙
- PR 설명에 변경 사항, 테스트 방법을 명시한다
- Self-review를 먼저 수행한다
- P1 코멘트는 반드시 수정 후 재요청한다
- P2 코멘트는 수정하거나 사유를 남긴다
